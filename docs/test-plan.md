# テスト計画書

## 1. テスト方針

- 実装フェーズに入る前にテスト観点を確定し、各フェーズで対応するテストを実施する
- 自動テストと手動テストを組み合わせ、品質を担保する
- 個人利用アプリであるため、負荷テストは省略し機能テストと結合テストを重点的に行う

---

## 2. テストレベル

| テストレベル | 説明 | 担当ツール |
|------------|------|----------|
| 単体テスト | モデル・サービスの個別ロジック検証 | RSpec（Rails） / Jest（Next.js） |
| 結合テスト | APIエンドポイントの入出力検証 | RSpec（Request Spec） |
| E2Eテスト | ブラウザ上での画面操作シナリオ検証 | 手動テスト（初期フェーズ） |
| インフラテスト | Terraform 適用後のリソース確認 | AWS CLI での確認 |

---

## 3. テスト対象一覧

### バックエンド（Rails）単体テスト

| テスト対象 | テスト観点 |
|-----------|----------|
| Recipe モデル | バリデーション（title必須・servings範囲・genre選択肢） |
| Ingredient モデル | バリデーション（name必須・recipe_id存在確認） |
| Instruction モデル | バリデーション（step_number重複不可・body必須） |
| Nutrition モデル | recipe_idのユニーク制約 |
| Tag モデル | nameのユニーク制約・バリデーション |
| ShoppingItem モデル | checked の初期値 false |
| NutritionCalculatorService | 食品マスタとの照合・計算ロジック |

### バックエンド（Rails）結合テスト（Request Spec）

| エンドポイント | テストケース |
|-------------|------------|
| GET /api/v1/recipes | 正常: レシピ一覧が返る |
| | 検索: q パラメータで絞り込める |
| | 検索: genre パラメータで絞り込める |
| | 検索: ingredient パラメータで絞り込める |
| | ページネーション: per_page, page が機能する |
| GET /api/v1/recipes/:id | 正常: 詳細データ（材料・手順・栄養情報含む）が返る |
| | 異常: 存在しないIDで404が返る |
| POST /api/v1/recipes | 正常: レシピが登録され201が返る |
| | 異常: titleなしで400が返る |
| | 異常: servingsが範囲外で400が返る |
| | 異常: 画像サイズ超過で422が返る |
| PATCH /api/v1/recipes/:id | 正常: 更新され200が返る |
| | 異常: 存在しないIDで404が返る |
| DELETE /api/v1/recipes/:id | 正常: 削除され204が返る |
| | 関連データ: 材料・手順・栄養情報も削除される |
| POST /api/v1/shopping_lists/:id/items | 正常（レシピ一括追加）: 材料が人数換算で追加される |
| | 正常（個別追加）: 指定した材料が追加される |
| PATCH /api/v1/shopping_lists/:id/items/:item_id | checked フラグが更新される |

### フロントエンド（Next.js）単体テスト

| テスト対象 | テスト観点 |
|-----------|----------|
| RecipeCard | タイトル・画像・ジャンル・調理時間が表示される |
| SearchBar | 入力値が onChange で反映される |
| FilterBar | フィルター選択で onFilter コールバックが呼ばれる |
| IngredientInput | 行追加・削除・入力値変更が正しく動作する |
| InstructionInput | 行追加・削除・入力値変更が正しく動作する |
| NutritionCard | 各栄養素の値が正しくフォーマットされて表示される |
| ImageUploader | ファイル選択でプレビューが表示される |

## 4. E2E テストシナリオ（手動）

### シナリオ 1: レシピ登録フロー（ゴールデンパス）

```
1. トップページを開く
2. 「+ 新規登録」ボタンをクリック
3. タイトル「肉じゃが」を入力
4. ジャンル「和食」を選択
5. 人数「4」を入力、調理時間「30」を入力
6. 材料を3件追加（名前・分量・単位）
7. 手順を3件追加
8. 「登録する」ボタンをクリック
9. レシピ詳細画面に遷移し、入力内容が表示されることを確認
10. トップページに戻り、一覧にカードが表示されることを確認
```

### シナリオ 2: レシピ検索フロー

```
1. トップページの検索バーに「じゃがいも」と入力
2. 絞り込み結果として該当レシピのみが表示されることを確認
3. 検索バーをクリアし、全件が表示されることを確認
4. ジャンルフィルター「和食」をクリック
5. 和食レシピのみが表示されることを確認
```

### シナリオ 3: 買い物リスト追加フロー

```
1. レシピ詳細画面を開く
2. 人数を「2」に変更
3. 「買い物リストに追加」ボタンをクリック
4. 買い物リスト画面に遷移し、材料が2人前換算で追加されていることを確認
5. 材料のチェックボックスをONにし、購入済み表示になることを確認
6. 「購入済みを削除」ボタンで、チェック済みアイテムが削除されることを確認
```

### シナリオ 4: レシピ編集・削除フロー

```
1. レシピ詳細画面を開く
2. 「編集」ボタンをクリック
3. タイトルを変更して「更新する」をクリック
4. 詳細画面に戻り、タイトルが変更されていることを確認
5. 「削除」ボタンをクリック
6. 確認ダイアログで「削除する」をクリック
7. トップページに遷移し、該当レシピが一覧から消えていることを確認
```

### シナリオ 5: エラーハンドリング確認

```
1. レシピ登録フォームでタイトルを空にして「登録する」をクリック
2. エラーメッセージ「タイトルは必須です」が表示されることを確認
3. 5MB超の画像ファイルをアップロードしてエラーメッセージを確認
4. 存在しない URL（/recipes/99999）にアクセスし、404エラー画面を確認
```

## 5. インフラ確認チェックリスト

Terraform `apply` 後に以下を AWS CLI で確認する：

```bash
# VPC・サブネット確認
aws ec2 describe-vpcs --filters "Name=tag:Name,Values=recipe-app-vpc"
aws ec2 describe-subnets --filters "Name=tag:Project,Values=recipe-app"

# EC2 起動確認
aws ec2 describe-instances --filters "Name=tag:Name,Values=recipe-app-ec2"

# RDS 起動確認
aws rds describe-db-instances --db-instance-identifier recipe-app-db

# S3 バケット確認
aws s3 ls | grep recipe-app

# CloudFront 確認
aws cloudfront list-distributions | grep recipe-app
```

## 6. テスト実施タイミング

| タイミング | 実施テスト |
|-----------|----------|
| 各機能の実装完了後 | 該当機能の単体テスト・結合テスト |
| 全機能の実装完了後 | E2Eテスト全シナリオ |
| インフラ構築後 | インフラ確認チェックリスト |
| 本番デプロイ後 | E2Eテスト（スモークテスト）全シナリオを本番環境で実施 |
| 機能更新・バグ修正後 | 該当機能 + 関連機能の回帰テスト |
